# Commit-Reveal Protocol

This document specifies the commit-reveal mechanism used to prevent result copying and front-running in DFPN.

## Problem Statement

Without commit-reveal, malicious workers could:

1. **Copy results**: Wait for other workers to submit, then copy their answers
2. **Front-run**: See pending transactions and submit the same result faster
3. **Collude**: Coordinate responses after seeing early submissions

These attacks undermine consensus quality and allow free-riding on honest workers.

## Protocol Overview

The commit-reveal protocol splits result submission into two phases:

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   COMMIT    │────▶│    WAIT     │────▶│   REVEAL    │
│   Phase     │     │   Period    │     │   Phase     │
└─────────────┘     └─────────────┘     └─────────────┘
     │                    │                    │
     │ Submit hash        │ All commits        │ Submit actual
     │ of result          │ are hidden         │ result + salt
     │                    │                    │
     ▼                    ▼                    ▼
 Commitment           No one can           Results are
 is binding           see results          verified
```

## Phases

### Phase 1: Commit

Workers submit a cryptographic commitment (hash) of their result without revealing the actual analysis.

**Commitment formula:**
```
commitment = SHA256(result_bytes || salt || worker_pubkey || request_id)
```

Where:
- `result_bytes`: Serialized `AnalysisResult` struct
- `salt`: 16 random bytes generated by the worker
- `worker_pubkey`: Worker's Solana public key
- `request_id`: The analysis request PDA address

**On-chain state after commit:**
```rust
struct ResultCommit {
    request: Pubkey,           // Analysis request
    worker: Pubkey,            // Worker who committed
    commitment: [u8; 32],      // SHA256 hash
    commit_slot: u64,          // Solana slot when committed
    revealed: bool,            // Whether reveal has occurred
}
```

### Phase 2: Reveal

After the commit window closes, workers reveal their actual results along with the salt used.

**Verification:**
```rust
fn verify_reveal(commit: &ResultCommit, result: &AnalysisResult, salt: &[u8; 16]) -> bool {
    let mut hasher = Sha256::new();
    hasher.update(&result.try_to_vec().unwrap());
    hasher.update(salt);
    hasher.update(&commit.worker.to_bytes());
    hasher.update(&commit.request.to_bytes());

    let computed = hasher.finalize();
    computed.as_slice() == commit.commitment
}
```

**On-chain state after reveal:**
```rust
struct ResultReveal {
    request: Pubkey,
    worker: Pubkey,
    result: AnalysisResult,
    salt: [u8; 16],
    reveal_slot: u64,
    verified: bool,            // Commitment matched
}
```

## Timing Windows

```
Request Created
      │
      ▼
┌─────────────────────────────────────────────────────────┐
│                    COMMIT WINDOW                         │
│         Workers can submit commitments                   │
│                                                          │
│  Duration: min(deadline - buffer, max_commit_window)     │
│  Typical: 60-80% of time until deadline                  │
└─────────────────────────────────────────────────────────┘
      │
      ▼
┌─────────────────────────────────────────────────────────┐
│                    REVEAL WINDOW                         │
│         Workers must reveal their results                │
│                                                          │
│  Duration: deadline - commit_window_end                  │
│  Typical: 20-40% of time until deadline                  │
└─────────────────────────────────────────────────────────┘
      │
      ▼
   Deadline
      │
      ▼
┌─────────────────────────────────────────────────────────┐
│                   FINALIZATION                           │
│         Anyone can trigger result aggregation            │
└─────────────────────────────────────────────────────────┘
```

### Timing Parameters

| Parameter | Default | Governance Range |
|-----------|---------|------------------|
| `min_commit_window` | 30 seconds | 15-120 seconds |
| `max_commit_window` | 5 minutes | 1-30 minutes |
| `commit_ratio` | 0.7 | 0.5-0.9 |
| `reveal_grace_period` | 10 seconds | 0-60 seconds |

**Calculation:**
```
total_time = deadline - request_created
commit_window = min(total_time * commit_ratio, max_commit_window)
commit_window = max(commit_window, min_commit_window)
reveal_window = total_time - commit_window
```

## Implementation Details

### Generating Commitments (Worker Side)

```typescript
import { sha256 } from '@noble/hashes/sha256';
import { randomBytes } from 'crypto';

function createCommitment(
  result: AnalysisResult,
  workerPubkey: PublicKey,
  requestId: PublicKey
): { commitment: Uint8Array; salt: Uint8Array } {
  // Generate random salt
  const salt = randomBytes(16);

  // Serialize result (using borsh or similar)
  const resultBytes = serializeResult(result);

  // Build commitment input
  const input = Buffer.concat([
    resultBytes,
    salt,
    workerPubkey.toBuffer(),
    requestId.toBuffer()
  ]);

  // Compute hash
  const commitment = sha256(input);

  return { commitment, salt };
}
```

### Submitting Commit (On-chain)

```rust
pub fn commit_result(
    ctx: Context<CommitResult>,
    commitment: [u8; 32],
) -> Result<()> {
    let request = &ctx.accounts.request;
    let clock = Clock::get()?;

    // Verify commit window is open
    require!(
        clock.unix_timestamp < request.commit_deadline,
        ErrorCode::CommitWindowClosed
    );

    // Verify worker is registered and staked
    let worker = &ctx.accounts.worker;
    require!(worker.stake >= MIN_STAKE, ErrorCode::InsufficientStake);

    // Verify worker hasn't already committed
    require!(
        ctx.accounts.commit_account.commitment == [0u8; 32],
        ErrorCode::DuplicateCommit
    );

    // Store commitment
    let commit = &mut ctx.accounts.commit_account;
    commit.request = request.key();
    commit.worker = worker.key();
    commit.commitment = commitment;
    commit.commit_slot = clock.slot;
    commit.revealed = false;

    // Update request state
    let request = &mut ctx.accounts.request;
    request.commit_count += 1;

    Ok(())
}
```

### Submitting Reveal (On-chain)

```rust
pub fn reveal_result(
    ctx: Context<RevealResult>,
    result: AnalysisResult,
    salt: [u8; 16],
) -> Result<()> {
    let request = &ctx.accounts.request;
    let commit = &ctx.accounts.commit_account;
    let clock = Clock::get()?;

    // Verify reveal window is open
    require!(
        clock.unix_timestamp >= request.commit_deadline,
        ErrorCode::RevealWindowNotOpen
    );
    require!(
        clock.unix_timestamp < request.deadline,
        ErrorCode::RevealWindowClosed
    );

    // Verify commitment exists and hasn't been revealed
    require!(
        commit.commitment != [0u8; 32],
        ErrorCode::NoCommitment
    );
    require!(!commit.revealed, ErrorCode::AlreadyRevealed);

    // Verify commitment matches
    let computed = compute_commitment(
        &result,
        &salt,
        &commit.worker,
        &commit.request
    );
    require!(
        computed == commit.commitment,
        ErrorCode::InvalidCommitment
    );

    // Store revealed result
    let reveal = &mut ctx.accounts.result_account;
    reveal.request = request.key();
    reveal.worker = commit.worker;
    reveal.result = result;
    reveal.salt = salt;
    reveal.reveal_slot = clock.slot;
    reveal.verified = true;

    // Mark commitment as revealed
    let commit = &mut ctx.accounts.commit_account;
    commit.revealed = true;

    // Update request state
    let request = &mut ctx.accounts.request;
    request.reveal_count += 1;

    Ok(())
}
```

## Edge Cases

### Worker Commits but Doesn't Reveal

**Scenario**: Worker submits commitment but fails to reveal before deadline.

**Handling**:
- Commitment is marked as `revealed = false` after deadline
- Worker receives reputation penalty (missed deadline)
- Stake slash: 1-3% depending on frequency
- Result is excluded from consensus

```rust
// During finalization
if !commit.revealed && clock.unix_timestamp > request.deadline {
    apply_missed_deadline_penalty(&mut worker)?;
}
```

### Invalid Reveal (Commitment Mismatch)

**Scenario**: Worker reveals data that doesn't match their commitment.

**Handling**:
- Transaction fails with `ErrorCode::InvalidCommitment`
- Worker cannot participate in this request's rewards
- No additional penalty (they can't game the system this way)

### Late Commit

**Scenario**: Worker tries to commit after commit window closes.

**Handling**:
- Transaction fails with `ErrorCode::CommitWindowClosed`
- Worker can still participate in future requests

### No Commits Received

**Scenario**: No workers commit to a request before deadline.

**Handling**:
- Request marked as `Expired`
- Fee refunded to requester (minus small protocol fee)
- Request can be resubmitted with higher fee or longer deadline

### Single Commit

**Scenario**: Only one worker commits and reveals.

**Handling**:
- If `min_workers == 1`: Result is accepted
- If `min_workers > 1`: Request marked as `Incomplete`, partial refund
- Single result is stored but flagged as low-confidence

## Security Analysis

### Attack: Commitment Grinding

**Description**: Attacker tries many salt values to find a commitment that matches another worker's.

**Mitigation**: Including `worker_pubkey` in the commitment makes this computationally infeasible. Each worker's commitment is unique to them.

### Attack: Timing Analysis

**Description**: Attacker analyzes transaction timing to infer which workers are participating.

**Mitigation**:
- Workers can submit at any time during commit window
- Random delays can be added client-side
- Commitment doesn't reveal the result

### Attack: Result Inference from Commitment

**Description**: Attacker tries to brute-force the result from the commitment hash.

**Mitigation**:
- Salt adds 128 bits of entropy
- Result space is large (verdict + confidence + detections)
- SHA256 is computationally secure

### Attack: Selective Reveal

**Description**: Worker sees other reveals and decides whether to reveal based on consensus.

**Mitigation**:
- Not revealing incurs penalties (stake slash, reputation loss)
- Workers don't know others' results until after their own reveal
- Reveal order doesn't affect rewards

### Attack: Front-Running Reveals

**Description**: Attacker sees a reveal transaction in mempool and submits their own first.

**Mitigation**:
- Commitment is binding; attacker would need to have committed first
- Even if attacker reveals first, their result must match their earlier commitment
- No advantage to revealing earlier (all valid reveals are treated equally)

## Monitoring and Alerts

Operators should monitor for:

| Signal | Threshold | Potential Issue |
|--------|-----------|-----------------|
| Commit-to-reveal ratio | < 90% | Workers failing to reveal |
| Invalid reveal rate | > 1% | Bug or attack attempt |
| Timing concentration | > 50% in last 10% of window | Potential coordination |
| Single-worker requests | > 20% | Insufficient worker supply |

## Future Considerations

### Threshold Reveal

Instead of all-or-nothing reveals, use threshold cryptography:
- Workers submit encrypted results
- Reveal key is reconstructed when threshold of workers participate
- Resistant to worker dropout

### Zero-Knowledge Proofs

Allow workers to prove properties of their result without revealing it:
- Prove result matches commitment
- Prove result is well-formed
- Enable partial verification before full reveal

### MEV Protection

Integrate with MEV protection mechanisms:
- Private mempools for commit/reveal transactions
- Batch auctions for result submission
- Flashbots-style bundles on Solana (Jito)
